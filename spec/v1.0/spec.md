## Abstract

The AI Provenance Protocol (APP) defines an open, vendor-neutral format for recording, embedding, and verifying the provenance of AI-generated content. This specification describes the metadata schema, embedding modes, verification protocol, and conformance requirements for implementations.

APP enables any system producing or consuming AI-generated content to answer fundamental provenance questions in a machine-readable, interoperable way: was this content generated by AI, which model produced it, what inputs were used, and whether a human reviewed it.

## Status of This Document

This is a **draft** specification. It is subject to change based on community feedback and implementation experience. The specification will be finalised once at least three independent implementations have been validated against the conformance test suite.

## 1. Introduction

### 1.1 Purpose

AI-generated content is proliferating across every industry, yet there is no widely adopted, interoperable standard for communicating its provenance. The AI Provenance Protocol addresses this gap by defining:

- A **metadata schema** for recording provenance information about AI-generated content
- **Embedding modes** for transmitting provenance metadata alongside content
- A **verification protocol** for third-party confirmation of provenance claims

### 1.2 Scope

This specification covers provenance metadata for AI-generated text, structured data, code, and any content representable as JSON or transmittable via HTTP. It is designed to be extensible to other modalities.

APP does not define:
- How AI systems should be built, trained, or constrained
- Content moderation or quality assessment mechanisms
- Watermarking or steganographic detection methods
- Cryptographic signing of content (though it is compatible with such approaches)

### 1.3 Relationship to Other Standards

APP is designed to complement, not replace, existing standards:

| Standard | Relationship |
|----------|-------------|
| C2PA / Content Credentials | Complementary. C2PA covers image/video/audio with cryptographic manifests. APP covers text and structured data. The `extensions` namespace allows referencing C2PA manifests. |
| Model Context Protocol (MCP) | Complementary. MCP defines AI-to-tool connectivity. APP records provenance of AI output. MCP tool responses can include APP metadata. |
| Schema.org | Interoperable. APP metadata can be expressed alongside Schema.org `CreativeWork` properties. |
| W3C PROV | Compatible. APP can be mapped to the W3C PROV data model for systems that require general provenance ontologies. |
| IPTC AI metadata | Complementary. APP extends the provenance concept beyond photojournalism to all AI-generated content types. |

### 1.4 Design Principles

1. **Simplicity** — The core schema has five required fields. Implementations can be as simple as a JSON object.
2. **Embeddability** — APP metadata can be embedded inline, transmitted as HTTP headers, or linked via URI.
3. **Verifiability** — An optional verification protocol allows third parties to confirm provenance claims.
4. **Extensibility** — Domain-specific metadata is supported via namespaced extensions.
5. **Privacy-aware** — The schema records roles, not personal identifiers. Generation IDs are pseudonymous.
6. **Regulation-ready** — The schema maps directly to EU AI Act Article 50 obligations.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

| Term | Definition |
|------|-----------|
| **APP metadata** | A JSON object conforming to this specification that describes the provenance of AI-generated content |
| **Generation event** | A single invocation of an AI model that produces content |
| **Generator** | The system (platform + model) that produced the content |
| **Platform** | The software application or service that orchestrates the AI model |
| **Embedding** | The method by which APP metadata is associated with generated content |
| **Verification endpoint** | An HTTP endpoint that can confirm the existence and details of a generation event |
| **Content hash** | A cryptographic hash of the generated content, used for integrity verification |
| **Conformant implementation** | A system that correctly produces, embeds, or consumes APP metadata as defined in this specification |

## 3. Conformance

### 3.1 Conformance Levels

This specification defines three conformance levels:

#### Level 1: Producer
A system that generates APP metadata and associates it with AI-generated content. A conformant producer:

- MUST generate valid APP metadata objects conforming to the schema in Section 4
- MUST include all required fields
- MUST use at least one embedding mode defined in Section 10
- SHOULD include a `verification_uri` field
- SHOULD include a `content_hash` field

#### Level 2: Consumer
A system that reads and interprets APP metadata. A conformant consumer:

- MUST correctly parse APP metadata from all three embedding modes
- MUST validate the `app_version` field and handle unknown versions gracefully
- SHOULD verify provenance claims against the `verification_uri` when present
- MUST NOT reject content solely because APP metadata is present

#### Level 3: Verifier
A system that hosts a verification endpoint. A conformant verifier:

- MUST implement the Level 1 verification protocol (Section 11.1)
- SHOULD implement the Level 2 verification protocol (Section 11.2)
- MUST serve verification endpoints over HTTPS in production
- MUST NOT return generated content through the verification endpoint
- SHOULD implement rate limiting

### 3.2 Versioning

The `app_version` field uses Semantic Versioning ([SemVer](https://semver.org/)):

- **Major** version changes indicate breaking changes to the schema
- **Minor** version changes add new optional fields (backward-compatible)
- **Patch** version changes are editorial corrections

Consumers MUST accept metadata with a matching major version, regardless of minor or patch differences. Consumers SHOULD warn on unknown major versions rather than failing silently.

## 4. Core Metadata Schema

The APP metadata object is a JSON object that MUST contain the following fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `app_version` | `string` | REQUIRED | Protocol version in semver format (e.g., `"1.0.0"`) |
| `ai_generated` | `boolean` | REQUIRED | `true` if the content was generated or substantially modified by AI |
| `generator` | `object` | REQUIRED | Information about the generating system (see Section 5) |
| `generated_at` | `string` | REQUIRED | ISO 8601 timestamp of generation completion (UTC RECOMMENDED) |
| `generation_id` | `string` | REQUIRED | UUID v4 uniquely identifying this generation event |

The following fields are OPTIONAL:

| Field | Type | Description |
|-------|------|-------------|
| `inputs` | `object` | Description of inputs used for generation (see Section 6) |
| `review` | `object` | Human review status (see Section 7) |
| `usage` | `object` | Token/resource usage (see Section 8) |
| `verification_uri` | `string` (URI) | Endpoint where this generation can be verified |
| `parent_generation_id` | `string` (UUID) | Links to a previous generation event (e.g., reprompt, refinement) |
| `content_hash` | `string` | Hash of the generated content for integrity verification (see Section 12) |
| `extensions` | `object` | Namespace for domain-specific extensions (see Section 9) |

### 4.1 Field Specifications

#### `app_version`
- MUST be a valid semver string
- For this version of the specification: `"1.0.0"`

#### `ai_generated`
- MUST be `true` if any substantial portion of the content was generated by an AI model
- MUST be `true` if AI-generated content was used without meaningful human modification
- MAY be `false` if a human substantially rewrote the AI output (in which case the APP metadata serves as a record that AI was involved in the process)

#### `generated_at`
- MUST be an ISO 8601 datetime string
- SHOULD include timezone information
- UTC is RECOMMENDED (e.g., `"2026-03-06T14:22:00Z"`)

#### `generation_id`
- MUST be a UUID v4 (RFC 4122)
- MUST be unique per generation event
- Platforms MAY use UUID v7 for time-ordered identifiers, provided the format is valid UUID

#### `verification_uri`
- When present, MUST be a valid URI (RFC 3986)
- SHOULD use HTTPS in production
- The endpoint at this URI SHOULD conform to the verification protocol in Section 11

#### `parent_generation_id`
- When present, MUST be a valid UUID
- Used to establish provenance chains (e.g., when content is regenerated or refined)
- Consumers SHOULD NOT require resolution of parent generation IDs

#### `content_hash`
- Format: `"{algorithm}:{hex-encoded-hash}"`
- See Section 12 for hashing requirements

### 4.2 Minimal Valid Example

```json
{
  "app_version": "1.0.0",
  "ai_generated": true,
  "generator": {
    "platform": "example-platform",
    "model": "openai/gpt-4o"
  },
  "generated_at": "2026-03-06T14:22:00Z",
  "generation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### 4.3 Complete Example

```json
{
  "app_version": "1.0.0",
  "ai_generated": true,
  "generator": {
    "platform": "acme-content-engine",
    "platform_version": "2.3.1",

  },
  "generated_at": "2026-03-06T14:22:00Z",
  "generation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "parent_generation_id": "00112233-4455-6677-8899-aabbccddeeff",
  "content_hash": "sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  "verification_uri": "https://verify.acme.com/v1/verify",
  "inputs": {
    "type": "structured",
    "description": "Product images and attribute data",
    "items": [
      { "kind": "image", "ref": "https://example.com/images/product-001.jpg", "hash": "sha256:abc123..." },
```

## 5. Generator Object

The `generator` object describes the system that produced the content.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `platform` | `string` | REQUIRED | Identifier of the platform that orchestrated generation |
| `model` | `string` | REQUIRED | Identifier of the AI model used |
| `platform_version` | `string` | OPTIONAL | Version of the platform |

### 5.1 Platform Identifier

- SHOULD use a human-readable name or reverse-domain format
- Examples: `"my-saas-product"`, `"acme-corp/content-engine"`, `"com.example.generator"`
- MUST be consistent across all generations from the same platform

### 5.2 Model Identifier

- SHOULD use the format `"{provider}/{model-name}"` (e.g., `"anthropic/claude-sonnet-4"`, `"google/gemini-2.5-flash"`, `"openai/gpt-4o"`)
- For self-hosted or fine-tuned models, the provider MAY be the organisation name (e.g., `"acme-corp/custom-product-writer-v2"`)
- MUST identify the specific model used, not a generic category

## 6. Inputs Object

The `inputs` object describes what went into the generation. It is intentionally descriptive — recording what was used, not the full content — to avoid data leakage.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | `string` | REQUIRED | Category of input: `"text"`, `"structured"`, `"multimodal"`, `"code"`, or `"other"` |
| `description` | `string` | OPTIONAL | Human-readable description of the inputs |
| `items` | `array` | OPTIONAL | List of input items (see below) |
| `prompt_ref` | `string` | OPTIONAL | Identifier or hash of the prompt template used |

### 6.1 Input Items

Each item in the `items` array describes a single input source:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `kind` | `string` | REQUIRED | Type of input: `"text"`, `"image"`, `"data"`, `"code"`, `"audio"`, `"video"`, `"document"`, or `"other"` |
| `ref` | `string` | OPTIONAL | Reference to the input (URI, identifier, or field name) |
| `hash` | `string` | OPTIONAL | Content hash of the input for integrity |
| `fields` | `array` of `string` | OPTIONAL | For `"data"` kind: list of field names used |

### 6.2 Privacy Guidance

Platforms SHOULD NOT include full input content in the `inputs` object. The purpose is to describe the *type* and *nature* of inputs, not to reproduce them. Specifically:

- DO include: input types, field names, references, hashes
- DO NOT include: full prompt text, user-submitted content, personal data

## 7. Review Object

The `review` object records human oversight of AI-generated content. This directly supports the EU AI Act Article 50(4) human review exemption.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `human_reviewed` | `boolean` | REQUIRED | Whether a human has reviewed this content |
| `reviewer_role` | `string` | OPTIONAL | Role of the reviewer (not PII) |
| `reviewed_at` | `string` (ISO 8601) | OPTIONAL | Timestamp of review completion |
| `review_type` | `string` (enum) | OPTIONAL | Nature of the review (see below) |
| `review_notes` | `string` | OPTIONAL | Free-text notes from the reviewer |

### 7.1 Review Types

The `review_type` field MUST be one of:

| Value | Description |
|-------|-------------|
| `"approved_without_changes"` | Content reviewed and approved as-is |
| `"edited"` | Content reviewed and modified by the reviewer |
| `"rejected"` | Content reviewed and rejected |
| `"pending"` | Content is awaiting human review |

### 7.2 Reviewer Role

The `reviewer_role` field records the functional role of the reviewer, not their identity. Examples: `"editor"`, `"compliance_officer"`, `"content_manager"`, `"legal_reviewer"`.

Platforms MUST NOT include personally identifiable information (PII) in the `reviewer_role` field.

---

## 8. Usage Object

The `usage` object records resource consumption for the generation event.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `input_tokens` | `integer` | OPTIONAL | Number of input tokens consumed |
| `output_tokens` | `integer` | OPTIONAL | Number of output tokens produced |
| `total_tokens` | `integer` | OPTIONAL | Total tokens (input + output) |
| `duration_ms` | `integer` | OPTIONAL | Generation duration in milliseconds |

All fields are OPTIONAL. Platforms SHOULD include at least `total_tokens` and `duration_ms` when available.

## 9. Extensions

The `extensions` object provides a namespace for domain-specific metadata that is not part of the core specification.

### 9.1 Namespace Format

Extension keys MUST use reverse-domain notation to avoid collisions:

```json
{
  "extensions": {
    "com.example": { "custom_field": "value" },
    "org.c2pa": { "manifest_uri": "https://example.com/c2pa/manifest.cbor" },
    "io.modelcontextprotocol": { "server_name": "my-mcp-server", "tool_name": "generate_content" }
  }
}
```

### 9.2 Extension Rules

- Extension namespace keys MUST be valid reverse-domain strings
- Extensions MUST NOT redefine or conflict with core schema fields
- Consumers MUST ignore unrecognised extensions (fail-open)
- Extensions SHOULD be documented by their respective namespace owners

### 9.3 Well-Known Extensions

| Namespace | Owner | Purpose |
|-----------|-------|---------|
| `org.c2pa` | C2PA | Links to Content Credentials manifests |
| `io.modelcontextprotocol` | MCP | Model Context Protocol integration metadata |
| `org.schema` | Schema.org | Schema.org `CreativeWork` property mappings |

## 10. Embedding Modes

APP metadata can be associated with generated content using three modes. Producers MUST implement at least one mode. Consumers SHOULD support all three.

### 10.1 Mode 1: Inline Embedding

For JSON or structured data output, the APP metadata object is embedded under the `_ai_provenance` key at the top level of the output:

```json
{
  "title": "Premium Leather Wallet",
  "description": "Handcrafted from full-grain leather...",
  "_ai_provenance": {
    "app_version": "1.0.0",
    "ai_generated": true,
    "generator": {
      "platform": "example-platform",
      "model": "google/gemini-2.5-flash"
    },
    "generated_at": "2026-03-06T14:22:00Z",
    "generation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
  }
}
```

#### Rules
- The key MUST be `_ai_provenance` (underscore prefix indicates metadata)
- The `_ai_provenance` key MUST be at the top level of the JSON object
- Systems processing the JSON output SHOULD preserve the `_ai_provenance` key through transformations
- Systems that do not understand APP SHOULD ignore the `_ai_provenance` key
- When computing content hashes, the `_ai_provenance` key MUST be excluded (see Section 12)

### 10.2 Mode 2: HTTP Headers

For API responses where inline JSON embedding is impractical, APP metadata is transmitted as HTTP response headers:

```http
X-APP-Version: 1.0.0
X-APP-AI-Generated: true
X-APP-Generation-Id: a1b2c3d4-e5f6-7890-abcd-ef1234567890
X-APP-Generator-Platform: example-platform
X-APP-Generator-Model: google/gemini-2.5-flash
X-APP-Generated-At: 2026-03-06T14:22:00Z
X-APP-Verification-URI: https://verify.example.com/v1/verify
X-APP-Human-Reviewed: false
X-APP-Content-Hash: sha256:e3b0c44298fc1c14...
```

#### Header Mapping

| Header | Maps to | Required |
|--------|---------|----------|
| `X-APP-Version` | `app_version` | REQUIRED |
| `X-APP-AI-Generated` | `ai_generated` | REQUIRED |
| `X-APP-Generation-Id` | `generation_id` | REQUIRED |
| `X-APP-Generator-Platform` | `generator.platform` | REQUIRED |
| `X-APP-Generator-Model` | `generator.model` | REQUIRED |
| `X-APP-Generated-At` | `generated_at` | REQUIRED |
| `X-APP-Verification-URI` | `verification_uri` | OPTIONAL |
| `X-APP-Human-Reviewed` | `review.human_reviewed` | OPTIONAL |
| `X-APP-Content-Hash` | `content_hash` | OPTIONAL |
| `X-APP-Parent-Generation-Id` | `parent_generation_id` | OPTIONAL |

#### Rules
- All `X-APP-*` headers use the `X-APP-` prefix
- Header values MUST be ASCII-safe (no JSON encoding in headers)
- Boolean values are transmitted as `"true"` or `"false"` strings
- For full metadata (including extensions, inputs, review details), use Mode 3

### 10.3 Mode 3: Linked Metadata

A `Link` header or embedded URI points to a full APP metadata document:

```http
Link: <https://verify.example.com/v1/metadata/a1b2c3d4-...>; rel="ai-provenance"; type="application/app+json"
```

#### Rules
- The `rel` attribute MUST be `"ai-provenance"`
- The `type` attribute SHOULD be `"application/app+json"`
- The linked document MUST be a complete, valid APP metadata object
- The linked document MUST be served over HTTPS in production
- The linked document SHOULD be publicly accessible without authentication
- This mode works for any content type (HTML, PDF, images, etc.)

#### HTML Embedding

For HTML documents, a `<link>` element can be used:

```html
<link rel="ai-provenance" href="https://verify.example.com/v1/metadata/a1b2c3d4-..." type="application/app+json" />
```

## 11. Verification Protocol

The verification protocol allows third parties to confirm provenance claims without accessing the original content. Platforms implementing APP SHOULD expose a verification endpoint.

### 11.1 Level 1: Generation ID Lookup

```
GET {verification_uri}/{generation_id}
```

#### Success Response (Generation Exists)

```http
HTTP/1.1 200 OK
Content-Type: application/app+json

{
  "found": true,
  "app_version": "1.0.0",
  "ai_generated": true,
  "generator": {
    "platform": "example-platform",
    "model": "google/gemini-2.5-flash"
  },
  "generated_at": "2026-03-06T14:22:00Z",
  "review": {
    "human_reviewed": false
  }
}
```

#### Not Found Response

```http
HTTP/1.1 200 OK
Content-Type: application/app+json

{
  "found": false
}
```

#### Rules
- The endpoint MUST NOT require authentication
- The response MUST NOT include the generated content itself
- The response SHOULD include `generator`, `generated_at`, and `review` fields
- The response MAY include additional APP metadata fields
- The endpoint SHOULD return `200 OK` for both found and not-found cases (with `found` field distinguishing)
- Rate limiting (HTTP 429) is RECOMMENDED

### 11.2 Level 2: Content Hash Verification

```
POST {verification_uri}/match
Content-Type: application/json

{
  "content_hash": "sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  "content_type": "application/json"
}
```

#### Success Response (Match Found)

```http
HTTP/1.1 200 OK
Content-Type: application/app+json

{
  "matches": [
    {
      "generation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "ai_generated": true,
      "generator": {
        "platform": "example-platform",
        "model": "google/gemini-2.5-flash"
      },
      "generated_at": "2026-03-06T14:22:00Z"
    }
  ]
}
```

#### No Match Response

```http
HTTP/1.1 200 OK
Content-Type: application/app+json

{
  "matches": []
}
```

#### Rules
- `content_hash` MUST follow the format specified in Section 12
- `content_type` is OPTIONAL but RECOMMENDED to narrow matches
- The response MUST NOT include the generated content
- Rate limiting is STRONGLY RECOMMENDED to prevent enumeration attacks

## 12. Content Hashing

Content hashing enables integrity verification: confirming that content has not been modified since generation.

### 12.1 Hash Format

```
content_hash = "{algorithm}:{hex-encoded-hash}"
```

Example: `"sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"`

### 12.2 Supported Algorithms

| Algorithm | Identifier | Status |
|-----------|-----------|--------|
| SHA-256 | `sha256` | REQUIRED (default) |
| SHA-384 | `sha384` | OPTIONAL |
| SHA-512 | `sha512` | OPTIONAL |

Implementations MUST support `sha256`. Other algorithms MAY be supported.

### 12.3 Canonicalization

For JSON content, the hash MUST be computed over the **canonical** representation:

1. Remove the `_ai_provenance` key from the JSON object (if present)
2. Sort all object keys recursively in lexicographic order
3. Remove all insignificant whitespace
4. Encode as UTF-8

```
content_hash = "sha256:" + hex(SHA-256(canonicalize(content_without_app_key)))
```

For non-JSON content (plain text, HTML, etc.), the hash is computed over the raw bytes of the content.

## 13. Security Considerations

### 13.1 Metadata Integrity

APP metadata is not cryptographically signed in v1.0. This means:

- Metadata can be modified after generation by any party with access to the content
- The `content_hash` field provides tamper detection for the *content* but not for the *metadata itself*
- For high-assurance use cases, implementors SHOULD consider wrapping APP metadata in a cryptographic envelope (e.g., JWS, C2PA manifest)

Future versions of this specification MAY define a signing mechanism.

### 13.2 Verification Endpoint Security

- Verification endpoints MUST be served over HTTPS in production
- Endpoints SHOULD implement rate limiting to prevent:
  - Enumeration of generation IDs
  - Denial of service
  - Information harvesting
- Endpoints MUST NOT return generated content (only provenance metadata)
- Endpoints SHOULD log access for audit purposes

### 13.3 Spoofing

Without cryptographic signing, a malicious actor could:

- Attach APP metadata to non-AI content (false positive)
- Remove APP metadata from AI content (false negative)
- Modify APP metadata fields (e.g., changing the model identifier)

The verification protocol (Section 11) mitigates spoofing for the positive case: a third party can confirm that a claimed `generation_id` was actually produced by the stated platform.

### 13.4 Enumeration

Generation IDs (UUID v4) have 122 bits of entropy, making brute-force enumeration impractical. However, verification endpoints SHOULD still implement rate limiting.

## 14. Privacy Considerations

### 14.1 No Personal Data in Core Schema

The core APP schema intentionally avoids personal data:

- `reviewer_role` records a functional role, not a person's identity
- `generation_id` is a random UUID, not linked to a user
- `generator.platform` identifies a system, not a person

### 14.2 Input Privacy

The `inputs` object describes the *nature* of inputs, not their content. Implementations MUST NOT include personal data in the `inputs` object without explicit consent.

### 14.3 GDPR Considerations

- APP metadata may constitute "metadata" under GDPR if it can be linked to an individual
- Platforms operating in the EU SHOULD assess whether their APP metadata, in combination with other data, could identify a data subject
- The `extensions` namespace is the most likely location for privacy-sensitive data; extension authors SHOULD conduct their own privacy impact assessments

## 15. IANA Considerations

### 15.1 Media Type Registration

This specification defines the media type `application/app+json` for APP metadata documents.

- **Type name**: application
- **Subtype name**: app+json
- **Required parameters**: None
- **Optional parameters**: None
- **Encoding**: UTF-8
- **Security considerations**: See Section 13

### 15.2 Link Relation

This specification defines the link relation `ai-provenance`:

- **Relation Name**: ai-provenance
- **Description**: Links to AI provenance metadata for the associated resource
- **Reference**: This specification

### 15.3 HTTP Headers

## Appendix C: MCP Integration

The Model Context Protocol (MCP) defines how AI applications connect to external tools and data sources. APP complements MCP by recording provenance of the AI-generated output that results from those interactions.

### C.1 MCP Tool Responses with APP Metadata

MCP servers that generate content using AI models can include APP metadata in their tool responses:

```json
{
  "content": [
    {
      "type": "text",
      "text": "Generated product description..."
    }
  ],
  "_ai_provenance": {
    "app_version": "1.0.0",
    "ai_generated": true,
    "generator": {
      "platform": "product-description-mcp-server",
      "model": "anthropic/claude-sonnet-4"
    },
    "generated_at": "2026-03-06T14:22:00Z",
    "generation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
  }
}
```

### C.2 MCP Extension Namespace

MCP-related metadata can be stored in the `io.modelcontextprotocol` extension namespace:

```json
{
  "extensions": {
    "io.modelcontextprotocol": {
      "server_name": "product-description-server",
      "server_version": "1.2.0",
      "tool_name": "generate_description",
      "session_id": "mcp-session-123"
    }
  }
}
```

| Concern | MCP | APP |

| Provenance of AI output | Not covered | Primary purpose |
| Content verification | Not covered | Verification protocol |
